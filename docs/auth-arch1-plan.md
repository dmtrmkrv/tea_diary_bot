# План: durable-вход (Arch 1) + юр-плумбинг

> Свод решений по уходу от Telegram-зависимости входа. Обсуждено 26.06.2026.
> Связано: `docs/privacy-data-inventory.md`, память `project_auth_email_plan`, `project_public_launch`.

## Решение (Arch 1)
- **email+пароль (база) + Яндекс (опционально)** — durable-вход (работает в РФ без VPN).
- **Telegram уходит с экрана логина** → остаётся только как **разовый in-app claim** (перенос записей из бота, под VPN, один раз).
- Распределение записей в БД крайне концентрированное (один кит — 146, ещё один — 11, остальные ≤5, в основном 1) → общий claim делаем ради UX-впечатления, не ради объёма.

## Схема БД (аддитивная, без переноса данных)
- `users.id` = **суррогат** (existing строки сохраняют значение = старый telegram_id; новые — из sequence). FK дегустаций/коллекции уже смотрят на `users.id` → **данные не двигаем**, JWT `sub` не меняется (сессии целы).
- Новые колонки `users`: `telegram_id` (BIGINT UNIQUE NULL, бэкфилл = id), `email` (UNIQUE NULL, lower-case), `password_hash` (argon2id), `yandex_id` (UNIQUE NULL), `consented_at` (timestamptz NULL — когда юзер принял Политику; опц. + версия Политики) — для аудита согласия по 152-ФЗ.
- `CHECK` (хотя бы один идентификатор), `id` → autoincrement (`setval` = max+1), уникальные индексы.
- Один и тот же email в Яндекс- и email-аккаунте = пока разные аккаунты (без авто-связывания, v1).

## Claim (reassign)
Залогинен (email/яндекс, аккаунт пуст) → telegram-verify (**переиспускаем готовый HMAC** `/auth/telegram`) → найти бот-строку по `telegram_id` → reassign `tastings/tea_items/teaware` → текущий аккаунт → `set telegram_id` → guard (не заклеймлено / есть данные) → вернуть `N`. `seq_no` чистый (целевой аккаунт пуст). Фото едут каскадом по `tasting_id`.
**Дима:** разово админ выставляет email/пароль на его существующую строку (через register+claim идти не надо).

## Решения по входу / UX

**Правило гейтинга согласием (ФИНАЛ):** ПДн-согласие гейтит действие, ТОЛЬКО если оно может **создать аккаунт**. Единственное действие, которое аккаунт не создаёт — email-вход возвращающегося юзера → только оно без гейта.
- **Вкладка «Вход»:** email-вход — кнопка **всегда активна** (логин, аккаунт не создаёт); «Войти через Яндекс» — **гейтим** (find-or-create может создать).
- **Вкладка «Регистрация»:** email-регистрация И Яндекс — **обе гейтим** (создают аккаунт).
- Один чекбокс согласия на обеих вкладках; гейтит только перечисленные действия. Дивайдер между email-кнопкой и Яндексом.

**Механика гейта (Вариант A):** «выключенные» кнопки делаем визуально-disabled, но **кликабельными** (НЕ нативный `disabled` — он глотает клик). Клик без галочки → тряхнуть чекбокс + sonner-подсказка «Примите Политику…». Поставил галочку → кнопка активируется. Если нужно «держать подсказку до отметки» — инлайн-индикатором на чекбоксе, не висящим тостом.

**Яндекс — одно действие** (find-or-create): не делим на «вход/регистрация», единый ярлык **«Войти через Яндекс»** на обеих вкладках. Возвращающийся Яндекс-юзер тоже ставит галочку (до OAuth не знаем, новый он или нет → гейтим всегда, безопасно).

**Формы (bottom sheets):**
- Email-вход: email + пароль + **«Забыли пароль?»** (ведёт в «скоро»/заглушку, пока нет SMTP).
- Регистрация: email + пароль + **повтор пароля**. НЕ автоопределение (опечатка в пароле без сброса = лок-аут).
- Состояния ошибок — в макетах: «Аккаунт не найден» (email-вход несуществующего → подсказать регистрацию), валидация ввода (email без `@` и т.п.).

**Фиксация согласия:** при создании аккаунта (email-регистрация или новый Яндекс) сохраняем `consented_at` (+ версию Политики). Email-вход возвращающегося повторно не спрашивает — согласие дано при регистрации.

**Email-верификация и сброс пароля — отложены** (нет SMTP). Пока: «повтор пароля» + ручной сброс админом. Потом: доменная почта (SMTP) + DNS (SPF/DKIM/DMARC).

**Онбординг:** при ПЕРВОЙ регистрации (флаг `is_new`) → шторки «возможности приложения» → **последний слайд «Перенести записи из бота»** → claim-redirect → «Перенесено N»; «Пропустить» → пустое состояние.

## Cookie — consent-gated (важно!)
- Куки: **необходимые** (`token` — без согласия) vs **аналитические** (Метрика — нужно согласие).
- Баннер даёт **выбор**: `Принять всё` / `Только необходимые`. **Не блокирующий** оверлей — полоса снизу (нельзя ограничивать доступ в обмен на сбор данных).
- **`YandexMetrika` грузить только при `cookie_consent==='all'`**, реактивно (баннер пишет → компонент подхватывает без перезагрузки). При «Только необходимые» — не инициализировать. → после этого env Метрики можно вернуть на прод (без согласия не выстрелит).
- Позже: «Настройки cookie» в профиле (отзыв согласия на аналитику).
- Это ДВА разных согласия: ПДн-чекбокс на логине (обязательный) ≠ аналитика в баннере (опциональная).
- Текст баннера: «На сайте используются файлы cookie. Часть из них необходима для работы входа и сессии, часть — для аналитики (Яндекс.Метрика), которая помогает улучшать сервис. Аналитику можно отключить. Подробнее в Политике конфиденциальности.»

- `proxy.ts` — `/privacy` публичная (passthrough).
- Чекбокс-согласие (ПДн) на текущем логине + ссылка на `/privacy`.
- `components/CookieBanner.tsx` (выбор Принять всё / Только необходимые, localStorage `cookie_consent`) + рефактор `YandexMetrika` на consent-гейтинг.
- Профиль — ссылка «Политика конфиденциальности» (+ позже «Настройки cookie»).
**Backend:** нет обязательного.
**Гейт:** должен быть жив ДО возврата Метрики на прод и снятия паузы.

## Bundle B — Arch 1 (durable-вход, фазируемый)
**Backend:**
- Alembic-миграция (колонки users, бэкфилл, id→autoincrement, индексы, CHECK).
- Модель `User` — новые поля.
- Хеш паролей (`argon2-cffi`) + утилиты.
- `POST /auth/register`, `POST /auth/login` (email).
- Яндекс: `GET /auth/yandex/login-url` + callback (код → userinfo → find/create по `yandex_id`) + конфиг `YANDEX_*`.
- `POST /auth/telegram/claim` (для залогиненного; reassign, guard).
- Хелперы поиска по email/yandex_id (бот `get_or_create_user` не трогаем).
**Frontend:**
- Переделка логина (Email+Яндекс, вкладки Вход/Регистрация + повтор пароля, ошибки, согласие).
- `apiClient`: register/login/yandexLoginUrl/claim.
- Обработка Яндекс-callback.
- Онбординг-слайд «Перенести записи» + claim + результат; «Пропустить» → пустое состояние.
- Профиль — «Перенести записи из бота».
- (позже) сброс пароля.
**Порядок:** схема → email-вход → claim → Яндекс → SMTP/сброс.

## Макеты (отрисовать) + сценарии
**Экраны:** логин/регистрация (Email+Яндекс, вкладки, чекбокс; состояния: дефолт, согласие не отмечено=disabled, загрузка, ошибки); онбординг + слайд «Перенести записи»; claim (подсказка про Telegram/VPN, успех «Перенесено N», крайние случаи); пустое состояние; cookie-баннер; профиль (пункты переноса и Политики).
**Сценарии:** (1) новый без бота → email → пропуск → пусто; (2) новый → Яндекс; (3) бот-юзер → регистрация → перенос → Telegram → успех; (4) возврат (email/Яндекс); (5) забыл пароль (что показываем без сброса); (6) пропустил → перенос позже из профиля; (7) согласие не отмечено (вход заблокирован); (8) ошибки (email занят, неверный пароль, claim уже сделан / нет записей).

## Workflow
staging → тест на тестовом домене (под VPN) → фиче-ветка от `main` → PR → `main` → ручной редеплой прод-аппов (снять паузу `Frontend_Prod`). Прод сейчас на паузе до Bundle A (политика + cookie). Метрика на проде выключена (env убрана) — вернём после consent-гейтинга.
